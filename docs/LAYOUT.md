# 按鍵佈局

## 為什麼是這個排法

Codex Micro 那張圖真正的重點不是「按鈕能發指令」——那是巨集鍵盤做了二十年的事。重點是**方向反過來**：agent 問，你答。所以佈局的第一原則是：

> **問題在中間，答案在兩側。**

實機的樣子（不是 3×3 九宮格 —— 我一開始畫錯了）：

```
   ┌───────────┬───────────┬───────────┐
   │     ✓     │     ◐     │     ✗     │          ⬤  K2
   │  APPROVE  │  STATUS   │   DENY    │        （大旋鈕）
   ├───────────┼───────────┼───────────┤          Menu
   │     🎤    │     ➤     │     ■     │
   │   VOICE   │   SEND    │   STOP    │
   └───────────┴───────────┴───────────┘
        ●          ⚡          ●         ◉      ◉
      key6      DISPATCH     key8       K1      K3
                                       Model   Session
      ←── 無螢幕的實體小圓鈕 ──→        ←─ 小旋鈕 ─→
```

**6 顆 LCD 鍵**（3×2）、**3 顆無螢幕的小圓鈕**、**1 大 2 小共 3 顆旋鈕**（全部可轉可按）。

上排是**回答 agent**，下排是**對 agent 說話**。

四個決定：

**STATUS 放正中間。** 它是唯一一顆永遠在講話的鍵。眼睛掃過去先看到「agent 在幹嘛」，`✓` 和 `✗` 就在左右手邊。狀態 → 答案，視線不用跳。

**`✓` 和 `✗` 中間隔一整顆鍵。** 這是刻意的。誤觸 `✓` 的代價是放行一條你沒看清楚的破壞性指令，和誤觸 `✗` 完全不對等。Codex Micro 把兩顆排一起，我不跟——實體鍵沒有 undo。

**`■` STOP 在右下角。** 遠離 `✓`。而且它有雙重語意：有東西在等你時 = 拒絕並解除阻塞；沒有時 = 送 Escape 中斷 agent。

**`🎤` VOICE 和 `➤` SEND 相鄰。** 口述的流程是「講 → 看 → 送」，這兩顆是同一個動作的兩半，手指不用跳。SEND 是刻意存在的 —— 它讓「轉寫結果先給你看過」這個設計不需要你伸手去碰鍵盤。所以 `autoSubmit` 才敢預設關閉。

**`⚡` DISPATCH 被降級到場景鍵列。** 它就是那顆按三下開三個 agent、把整個 deck 卡死的鍵。放遠一點。

## 待命 vs 點亮

`✓` 和 `✗` **平常是暗的**（`#141416`，近乎全黑，無圖示）。只有真的有工具呼叫卡著等你，它們才會亮起來、描邊、跳出標籤。

這不是裝飾。一個永遠亮著 `✓` 的鍵，你會忘記它什麼時候是有意義的；一個平常全暗、突然亮起來的鍵，你會**看**它。這顆裝置的價值全押在「你有沒有真的看那一眼」上。

STATUS 鍵的五個狀態：

| 狀態 | 樣子 | 意思 |
|---|---|---|
| `idle` | 灰色月牙 | 沒事 |
| `thinking` | 藍色三點跳動（SVG `<animate>`）| agent 在跑 |
| `awaiting_approval` | 橘底＋對話框＋**工具名** | 在等你，鍵上直接寫著等什麼 |
| `done` | 綠勾 | 這輪結束 |
| `error` | 紅叉 | 出事了 |

`awaiting_approval` 時 STATUS 鍵上印的是 `tool_input` 摘要（`Bash` / `npm test`），所以**不用看螢幕就能決定**。這是整個設計的目的。

## 旋鈕

實機確認過的對應（哪顆實體旋鈕對應 slider 0/1/2 沒有任何文件，只能裝上去轉）：

三顆不是一樣的東西：**K2 是大旋鈕，K1／K3 是它旁邊的兩顆小的**（左 K1、右 K3）。全部可轉可按。

| 旋鈕 | 實體 | 模式 | 轉 | 按 |
|---|---|---|---|---|
| **K1** | 小（左）| `model` | 開 model 選單並移動 | 選定 |
| **K2** | **大** | `menu` | 在**已經開著**的選項裡移動 | Enter |
| **K3** | 小（右）| `session` | `Ctrl+Tab` / `Ctrl+Shift+Tab` | — |

**K2 和上排是同一件事的兩半。** 上排的 `✓` `✗` 回答二選一的問題；K2 回答**多選一**的問題 —— Claude 給你一串執行選項時，轉它、按它。它跟 `✓` `✗` 一樣，是「回答」而不是「命令」。

**K2 刻意什麼都不打開。** 它只在你眼前已有的東西裡移動。這個限制就是它的安全性來源 —— 它不可能把你導航到沒要求的地方。

**K1 走 UIA，不送快捷鍵。** 見下方。`mode`（權限模式）和 `effort` 也是同一套實作，PI 裡可以換。

⚠️ 全部都送給**前景視窗**。Claude Code 不在前面就會送錯地方。這是實作上的妥協，不是設計。

還有 `diff` 模式（盲送 Up/Down 捲動）留著，但它是為終端機版寫的 —— 桌面版的 diff 是獨立面板（`Ctrl+Shift+D`），效果不會如預期。

## 三顆小圓鈕（key 6-8）：沒有螢幕

OpenDeck 把它們當成一般按鍵暴露（`keys[6..8]`），連 `setImage` 都會接受 —— 但**它們沒有顯示器**，送過去的圖等於丟進黑洞。驅動 log 那句 `Setting image for button 8` 看起來一切正常，實際什麼都沒發生。

**這排除了任何「靠顯示才有意義」的功能。** `session`（唯一功能是顯示 session 數量）曾經被放在 key 8 —— 那是個純粹的謊言：按下去什麼也不做，而它想告訴你的東西你永遠看不到。已經拿掉。

適合放這裡的是**你靠位置記憶的動作**，就像它們原廠的上一頁／首頁／下一頁。目前只放了 DISPATCH（按了就跑，不需要看）。

## 未實作 / 已知限制

- **VOICE 鍵有三種模式**，選哪個取決於你講什麼語言：
  - **`local`（中文，預設）**：ffmpeg 錄音 → whisper.cpp 轉寫 → 貼進輸入框。**唯一聽得懂中文的模式** —— Claude Code 內建聽寫是英文限定，明確拒絕 zh-CN。全程本機，音訊不外流。
  - **`gui`（桌面版內建聽寫，英文）**：用 **UI Automation** 切換那顆聽寫按鈕（`lib/mic.ps1`）。桌面版**沒有麥克風快捷鍵**（`Ctrl+/` 清單裡沒有，CLI 的 `voice:pushToTalk` 也不適用），所以沒有按鍵可送 —— 只能操作按鈕本身。用**無障礙名稱**定位，不是座標。
  - **`cli`（終端機版，英文）**：送 `key` 給前景視窗，需先 `/voice tap`（`SendKeys` 送不出「按住」）。
  - **三個狀態全部是真的，沒有猜的**：`VOICE` → `REC` → `HEARING`。`local` 模式我們自己持有 ffmpeg 程序；`gui` 模式那顆按鈕暴露 `TogglePattern` 可以讀 `ToggleState`（plugin 每 2 秒輪詢，抓 Claude 因靜音自己停止的時刻）。`HEARING` 是必要的 —— whisper 要跑幾秒，鍵在那段時間變暗會像按壞了。
- **DISPATCH 是開新的 headless run**（`claude -p`），不是打字進你正在跑的 session。Claude Code 沒有對外的 prompt 注入 API，硬做只能模擬鍵盤，很脆。
  - 派出去的 run 帶 `--settings '{"hooks":{}}'`，**刻意不受這個 deck 管**。否則它會卡在等一個沒人看得到的批准 → 死鎖。
  - 同時只跑一個。按三下不會開三個 agent。

- **Deck 是單 session 的。** 狀態機只有一個 pending 欄位。兩個 session 同時要批准時，新的會把舊的擠成 `defer`。這不是 bug，是為了防卡死的取捨 —— 但代價是**這台裝置一次只服務一個 session**。全域掛 hook 而同時開很多 session，會讓它們互相踩。
- **旋鈕不能靠 OpenDeck UI 綁定 —— 已證實，也已繞過。** OpenDeck 的 UI 確實不讓你把 action 指派給 encoder 事件。解法是 `scripts/apply-profile.mjs` 直接寫 profile JSON，實測三顆旋鈕都掛得上（`sliders: diff session effort`）。所以這是 UI 的限制，不是韌體或協定的限制。
